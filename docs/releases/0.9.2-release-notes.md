# AssegaiPHP Core 0.9.2

Released: 2026-06-21

`0.9.2` is a routing, module-resolution, and rendering polish release for Core.

This release focuses on correctness in real application structures: nested modules, shared imported modules, host-aware runtime routes, handled exception logging, and web component document rendering. It also includes small static-asset fixes that prevent broken references in generated pages.

## Highlights

### More reliable nested and shared module resolution

Core now handles repeated imports of the same module class more carefully. This matters for applications where a module is imported from more than one branch of the module tree.

The injector and module manager now keep track of direct parent relationships instead of relying on whichever parent was seen first during traversal. That makes provider visibility checks less dependent on import order and prevents shared child modules from accidentally gaining access through an unrelated parent branch.

Child modules can still consume providers that a parent module explicitly exports. Private providers remain private.

### Safer controller activation in nested module graphs

Router activation now tracks visited modules while walking imports. This prevents cyclic or repeated module graphs from degrading into repeated traversal when resolving a controller for a request.

The selected controller still follows the most-specific route match rules, but traversal now avoids revisiting the same module branch during one activation pass.

### Host-aware runtime routing improvements

Host route patterns now understand constrained dynamic labels such as:

```php
#[Controller(host: ':vendor_runtime_slug<slug>.runtime.localhost')]
```

The captured host parameter is now exposed under the expected parameter name, and the configured constraint is applied before the controller is considered a match.

Wildcard route handlers now expose the consumed trailing path as a route parameter under the `'*'` key:

```php
#[Get('*')]
public function handle(
    #[HostParam('vendor_runtime_slug')] string $vendorRuntimeSlug,
    #[Param('*')] string $vendorPath,
): string {
}
```

Normal dynamic route parameters such as `:id` remain segment-scoped. A route like `/users/:id` does not match `/users/1/profile`.

Middleware route matching and OpenAPI placeholder extraction now understand wildcard paths consistently with the router.

### Better handled-exception logging

HTTP and Whoops exception handlers now log handled exceptions consistently.

Client-side HTTP errors are logged at debug level, while unexpected server-side failures are logged as errors. Production-facing error rendering remains sanitized; this change improves observability without changing the public error response contract.

### Web component and document rendering polish

The default template engine now handles structured document props more predictably while preserving support for legacy raw head props.

This improves document-level rendering for metadata, head assets, body script tags, import-map script tags, web component bundle tags, and htmx script configuration. Template helper registration now also avoids relying on `method_exists()` for dynamic context methods.

### Public asset and bundle URL fixes

Static asset handling now reports the correct MIME type for `.webp` files.

Web component bundle URLs are no longer emitted merely because web components are marked as enabled. Local bundle paths must exist, while fully-qualified remote URLs remain valid. This avoids rendering broken public asset references in pages that do not have a generated local bundle yet.

## Upgrade Notes

No application-wide migration is required.

For runtime-style catchall controllers, prefer binding the wildcard path with `#[Param('*')]` on a `#[Get('*')]`, `#[Post('*')]`, or equivalent handler. Do not rely on a normal `:param` segment to capture multiple path segments; dynamic params remain intentionally segment-scoped.

If an application has ambiguous shared module imports, review provider exports and ensure only intentionally public providers are exported from the parent modules that child modules should see.

If a project configures a local web component bundle path, make sure the bundle is present before expecting Core to render the module script tag. Remote bundle URLs continue to render without a local file check.

## Validation

This release adds or updates regression coverage for:

- cyclic and repeated module traversal during controller activation
- child module access to parent exports and rejection of parent private providers
- shared child modules imported from multiple branches
- constrained host route parameters
- wildcard route path parameter binding
- normal dynamic route params staying segment-scoped
- wildcard route handling in middleware and OpenAPI metadata
- handled exception logging behavior
- structured and legacy document props in the default template engine
- missing web component bundles and `.webp` MIME type handling
