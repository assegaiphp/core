# 0.9.0 Upgrade Notes Draft

## Path

- From: `0.8.x`
- To: `0.9.0`

## Automatic

- `assegai update` should bring the tied first-party package line forward in a normal Assegai workspace.
- Existing `#[Entity(database: ...)]` declarations remain supported.
- Existing `#[Entity(engine: ...)]`, `#[Entity(schema: ...)]`, and `#[Entity(withRowId: ...)]` declarations remain supported for compatibility.

## Manual

- Review ORM configuration for the SQL driver you actually use and make sure the matching PDO extension is enabled.
- If you are adopting the refreshed entity metadata style, prefer `dataSource: 'name'` over `database: 'name'` on new entities.
- If an entity needs SQL-only storage options such as schema qualification, engine selection, or SQLite `WITHOUT ROWID`, prefer `#[SqlEntityOptions(...)]` on new code.
- If you already use the older `Entity(...)` storage fields, you do not need to rewrite them immediately for `0.9.0`.

## Verification

- Run the app test suite after the upgrade.
- Run the ORM driver lane you actually depend on.
- Verify schema generation or migrations against a real development database before calling the upgrade complete.

## Notes

`0.9.0` keeps the old entity storage fields as compatibility surface, but the long-term direction is a cleaner backend-neutral metadata core with SQL-specific storage concerns moved out of the base entity attribute.
