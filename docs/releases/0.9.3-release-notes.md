# AssegaiPHP Core 0.9.3

Released: 2026-06-23

`0.9.3` is a routing, middleware, OpenAPI, and provider-diagnostics polish release for Core.

This release tightens compatibility around dynamic route and host patterns, keeps middleware route matching aligned with the router, and improves the diagnostics developers see when a provider is blocked by module export rules.

## Highlights

### Shared route-pattern parsing

Core now routes more consistently through a shared route-pattern parser.

The router and middleware matcher now understand the same route parameter forms, including constrained parameters and legacy hyphenated parameter names. This keeps controller dispatch and middleware matching from disagreeing about routes such as:

```php
#[Get('accounts/:tenant-id')]
```

Normal dynamic route parameters remain segment-scoped. A route like `users/:id` matches `users/42`; it does not consume `users/42/posts`.

### Dynamic host compatibility

Host patterns now preserve existing dynamic-host behavior while supporting constrained placeholders.

Existing hyphenated host parameter names continue to work:

```php
#[Controller(host: ':tenant-id.example.com')]
```

Constrained host parameters are also supported:

```php
#[Controller(host: ':tenant<slug>.example.com')]
```

The runtime extracts the expected host parameter name, applies the constraint, and avoids treating constraint syntax as part of the parameter value.

### OpenAPI server URL fixes for dynamic hosts

Generated OpenAPI documents now render dynamic host server URLs more accurately.

Constraint suffixes are stripped from server variables, and trailing ports remain outside the placeholder. For example, dynamic host patterns now render as URLs shaped like:

```text
http://{tenant}.example.com
http://{tenant}:8080
http://api.{tenant}:8080
```

This prevents generated docs and clients from treating constraints or ports as literal parts of the variable name.

### Middleware matching aligned with route names

Middleware route targets now use the same parsing behavior as the router for dynamic route names.

This keeps `forRoutes()` and explicit route targets working when a controller path uses a hyphenated parameter name or a constrained parameter. Middleware can continue to be bound by controller class, method, and route target without route syntax drift.

### Clearer provider export failures

Provider visibility errors now include clearer context when a provider cannot be resolved because the importing module does not export it.

This does not loosen module boundaries. Private providers remain private, and imported providers still need to be exported intentionally. The change makes misconfigured module graphs easier to diagnose.

## Upgrade Notes

No application-wide migration is required.

Applications using dynamic host parameters with hyphenated names should continue to work. Applications using constrained dynamic hosts should now see cleaner runtime matching and cleaner OpenAPI server URLs.

If an application has tests that assert exact provider-resolution error text, update those assertions for the more descriptive export-visibility message.

Route parameters are still segment-scoped. Use wildcard routes when a handler intentionally needs to capture a trailing path.

## Validation

This release adds or updates regression coverage for:

- constrained route parameters and rejected constraint mismatches
- legacy hyphenated route parameter names
- middleware matching for hyphenated and constrained route parameters
- dynamic host placeholders with constraints
- dynamic host placeholders with trailing ports
- OpenAPI server URL rendering for constrained and port-bearing host patterns
- provider access failures across module export boundaries
