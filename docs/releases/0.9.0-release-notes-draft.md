# AssegaiPHP Core 0.9.0

0.9.0 is the ORM stability rewrite release.

This milestone is about making the first-party data story honest, cross-dialect, and predictable. The ORM is no longer a mostly-MySQL path with partial alternatives around it. 0.9.0 hardens the SQL-family implementation across MySQL, MariaDB, PostgreSQL, SQLite, and MSSQL, and it aligns the guides with the way the code now actually works.

## Highlights

### A genuinely dialect-aware SQL family

The largest change in 0.9.0 is the query-builder and SQL-generation rewrite.

The ORM now routes create, drop, alter, rename, select, insert, update, delete, describe, truncate, and related statement building through dialect-aware roots instead of assuming one SQL flavor everywhere and patching around the differences later.

That means:

- MySQL and MariaDB keep their shared family behavior where it makes sense
- PostgreSQL and SQLite can express the syntax they actually need
- MSSQL now exists as a real SQL-family sibling instead of a placeholder path
- shared SQL builders are cleaner and less MySQL-shaped internally

### MSSQL is now a real supported path

0.9.0 closes the biggest dialect gap by bringing SQL Server into the supported family.

That includes:

- an MSSQL query-builder surface
- SQL Server identifier quoting and column rendering
- SQL Server-specific create, drop, describe, and rename behavior
- a real composer test:mssql lane
- live integration coverage against a running SQL Server instance

### Safer and more consistent query execution

Security-sensitive SQL paths have been tightened as part of the rewrite.

In particular:

- raw query execution now consistently prefers prepared statements
- emulated prepares are disabled where they should not be leaned on
- public SQL-builder paths were hardened while the dialect refactor was underway
- the ORM test matrix now covers more of the edge cases that used to hide behind one driver path

### Entity metadata is moving in a cleaner direction

0.9.0 also starts the transition toward a more backend-neutral metadata model.

The biggest visible changes are:

- entity driver no longer defaults to MySQL silently
- dataSource is now the cleaner public alias for entity-level source selection
- SQL-only storage concerns now have a dedicated companion attribute named SqlEntityOptions

Existing Entity declarations that use database, engine, schema, or withRowId still work in 0.9.0. They remain as compatibility surface while the project moves toward a cleaner metadata split.

### Documentation now matches the shipped behavior better

The ORM guides now reflect:

- real driver support, including MSSQL
- named data_source and dataSource terminology instead of overly SQL-shaped wording where it no longer fits
- the current query-builder and dialect story
- the SqlEntityOptions direction for SQL-only storage metadata

## What did not change

- The legacy entity storage fields are not removed in 0.9.0. They remain supported for compatibility.
- The ORM is still centered on the SQL family today. Non-SQL backends are not part of this milestone.
- This release does not claim perfect parity across every possible dialect edge case.
- The broader long-term metadata redesign is not finished here. 0.9.0 starts that transition without forcing a breaking rewrite.

## Upgrade notes

If you are upgrading from 0.8.x:

- Existing entity metadata should continue to work without an immediate rewrite.
- For new code, prefer dataSource on Entity instead of database.
- For new SQL-only storage options, prefer SqlEntityOptions instead of putting those options directly on Entity.
- If you plan to use MSSQL, install and enable the matching PHP SQL Server extensions and validate against a real SQL Server instance.
- Re-run the ORM driver lane that matches your actual database after upgrading.

## What comes next

0.9.0 does not finish the entire ORM journey. It clears the main stability and honesty work so the next phase can aim higher.

The next important direction is making entity and attribute metadata more backend-neutral overall, while keeping SQL-family support strong and predictable.
