# AssegaiPHP Core 0.8.2

`0.8.2` is a security and hardening patch release on the `0.8.x` line.

It tightens public asset serving, closes a view template path traversal vector, normalizes the web component props helper name, and corrects config documentation.

No new features are introduced. No breaking changes are made to normal application behavior.

## What changed

### Hardened public asset serving

The logic that serves files from the `public/` directory now enforces a strict allowlist of permitted file extensions.

Previously, the extension check was loose enough that requests for `.phps` files and hidden files could reach the filesystem in ways that were not intended.

After this change:

- only a defined set of safe extensions can be served from `public/`
- `.phps` files are explicitly blocked
- hidden files are blocked except for the `.well-known/` directory, which is needed for ACME certificate challenges and similar well-known protocol paths

This reduces the attack surface for accidental source exposure through the public directory.

### View template path traversal protection

`View.php` now resolves template paths using dedicated path resolution methods that confirm the resolved path is inside the expected views or components directory before loading it.

Previously, a crafted template name could have resolved to a file outside the views tree.

After this change, any template path that resolves outside the permitted directory is rejected. This closes a directory traversal vector in template loading.

### Normalized web component props helper

The web component props helper has been standardized across the framework.

The old names `web_component_props` (global helper function) and `ctx.webComponentProps` (template context variable) are replaced by:

- `wc_props` as the global helper function
- `ctx.wcProps` as the template context variable

All documentation and internal template engine references have been updated to the new names.

See the upgrade notes for what this means for existing templates.

### Config documentation correction

Documentation for getting started and building a feature now correctly states that database settings are written to and read from `config/secure.php` rather than `config/default.php`, and explains the precedence of `secure.php` in the configuration cascade.

## What did not change

- The default PHP runtime behavior is unchanged
- The OpenSwoole runtime path is unchanged
- No database, ORM, routing, or module behavior changed
- No dependency versions changed

## Verification

- run `assegai serve` and confirm normal request handling
- check that static asset requests for expected file types still resolve correctly
- check that requests for `.phps` files or hidden paths outside `.well-known/` return a non-200 response
- if your app uses web components with the old `web_component_props` helper, update those calls before upgrading

## What comes next

The next milestone is `0.9.0`, focused on ORM stability:

- hardening MySQL support
- properly implementing SQLite as a supported target
- bringing PostgreSQL support up to the same level
- making the query builder and SQL generation properly dialect-aware
