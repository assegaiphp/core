# AssegaiPHP Core 0.8.2

`0.8.2` is the public asset allowlist hardening, view path traversal fix, and web component API cleanup release.

This is a focused security patch on top of `0.8.1`. It closes two remaining gaps in how the framework resolves file paths — one in public asset serving and one in view template loading — and cleans up an inconsistency in the web component props helper naming that was left over from earlier development.

## Highlights

### Extended public asset allowlist

`0.8.1` introduced `resolvePublicResourcePath()` to replace the unsafe `require_once` path. `0.8.2` tightens the extension filtering it applies.

Before `0.8.2`, the allowlist check did not explicitly cover `.phps` files, and the blanket rule against hidden files did not have a carve-out for the `.well-known/` directory path that ACME certificate challenges and similar protocols require.

`0.8.2` corrects both gaps:

* `.phps` files are now explicitly blocked at the extension level
* hidden files are still rejected, except for paths inside `.well-known/`, which are now passed through correctly
* the permitted extension set is now a tighter, explicit allowlist rather than a loose exclusion list

New unit tests cover the `.phps` block and the `.well-known` exception directly.

### View template path traversal protection

`View.php` previously assembled template file paths by joining a base directory string with a caller-supplied template name. A crafted template name containing `..` segments could resolve to a file outside the views or components directory tree.

`0.8.2` replaces the raw path join with dedicated resolution methods that:

* resolve the full path using `realpath`-style resolution
* confirm the resolved path is inside the permitted views or components directory before loading it
* reject any template path that escapes the directory boundary, regardless of how many traversal segments the name contains

A new unit test confirms that a sibling file outside the views directory cannot be loaded as a template.

### Web component props helper rename

The `web_component_props()` global helper function and the `ctx.webComponentProps` template context variable have been renamed for consistency with the rest of the helper naming conventions:

* `web_component_props(...)` → `wc_props(...)`
* `ctx.webComponentProps` → `ctx.wcProps`

All references inside the template engine, the global functions file, and the frontend and pages-and-components documentation have been updated to the new names.

If your templates or PHP code use the old names, see the upgrade notes below.

### Config cascade documentation correction

The getting-started guide and the building-a-feature guide previously described database settings as going into `config/default.php`. That was incorrect. Database credentials belong in `config/secure.php`, which takes precedence over `config/default.php` in the configuration cascade.

Both guides have been corrected to reflect that, and the cascade precedence rule is now stated explicitly.

## What did not change

* The OpenSwoole runtime path was not modified in this release
* No public API surface changed beyond the `wc_props` rename
* No configuration keys changed
* No database, ORM, routing, or module behavior changed
* The `0.8.0` and `0.8.1` upgrade paths are still valid — nothing in `0.8.2` changes those paths

## Upgrade notes

If you are upgrading from `0.8.1`:

* If your templates or PHP code call `web_component_props(...)`, rename those calls to `wc_props(...)`.
* If your templates reference `ctx.webComponentProps`, rename those references to `ctx.wcProps`.
* No other code changes are required for a normal upgrade from `0.8.1`. Public asset and view path changes are transparent to application code.
* If your application deliberately serves files with non-standard extensions from the `public/` directory, confirm those extensions are in the permitted allowlist after upgrading. Requests for extensions outside the allowlist will no longer be served.
